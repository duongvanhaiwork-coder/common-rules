# Shared Docs And Change Notes

## When To Use
- Any change affecting external behavior, contracts, or operations.
- Any breaking change or migration-relevant update.

## Checklist
1. Identify behavior or contract changes introduced.
2. Mark breaking changes explicitly and describe impact.
3. Add migration notes for schema/API/config transitions.
4. Add concise release note bullets focused on user impact.
5. Ensure docs reflect new defaults, limitations, or caveats.
6. Link verification context for claimed behavior updates.

## Output Format
- Behavior/contract changes
- Breaking-change notes
- Migration notes
- Release-note summary
